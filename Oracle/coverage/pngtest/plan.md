# Power Query Evaluator Test Suite: Image-Decoder Strategy

**Audience:** the Claude instance owning the Rust Power Query evaluator project (including `BinaryFormat.*` implementation).
**Author context:** produced in a planning conversation with the project owner. The owner has 45 years of programming experience and prefers discussion-first, code-on-request. This document is the result of that discussion; treat it as a design proposal to integrate, not a finalised spec. Push back where it's wrong.

---

## TL;DR

We want a test suite for the Rust Power Query / M evaluator that proves end-to-end correctness across `BinaryFormat.*`, numeric ops, list/record/table construction, error handling, and host bindings like `Binary.Decompress`. The proposed approach uses **image decoding as the integration test**, because:

- A correctly decoded image produces a single deterministic pixel hash → trivial pass/fail.
- Decoders exercise a wide cross-section of M features in a single program.
- Failures are diagnostic *and* eyeball-able (wrong image = something visibly wrong).

Two complementary tracks:

1. **PNG decoder** in M — realism track. Uses the PngSuite conformance corpus. Exercises `BinaryFormat.*`, `Binary.Decompress(Compression.Deflate)`, CRC32, per-scanline filter arithmetic, bit-depth unpacking, and optional Adam7 interlacing.
2. **PQTI (Power Query Test Image)** — a custom format designed *adversarially* against the evaluator. Each file isolates one or two M features with embedded expected results and a final pixel hash. Generated by a Rust tool; decoded by an M decoder.

Expected results are **cross-checked** between the Rust generator (computed from first principles) and a reference Power Query evaluator (Excel desktop or Power BI). Disagreement between those two is itself a finding worth investigating.

---

## Why this strategy

Standard unit tests check individual functions. They don't catch interaction bugs: laziness composing with error propagation, numeric coercion interacting with list materialisation, `BinaryFormat.Choice` dispatch failing only when nested three deep. Image decoders happen to compose exactly the operations that expose those interactions, and they produce one bit-exact output that's easy to compare.

PNG was effectively designed for conformance testing — chunked structure with CRCs, well-defined filter algorithms, an existing reference suite (PngSuite) maintained for decades. Wrapping it in M gives realism without us having to invent the test corpus.

PQTI fills the gap PNG doesn't cover: targeting specific evaluator features in isolation so a failure is diagnostic. PNG tells you "something is broken"; PQTI tells you "list recursion at depth 4096 is broken".

## Part 1: PNG Decoder Track

### Scope

**In:** baseline PNG decoding sufficient to pass the bulk of PngSuite.

**Out (initially):** APNG, EXIF chunks, sRGB/iCCP colour management (treat as opaque), text chunks beyond reading their lengths.

### Coverage progression (ordered by implementation effort vs evaluator surface gained)

Implement and test in this order; each stage must pass before the next is started.

1. **Greyscale 8-bit non-interlaced, filter type 0 only.** Minimum viable PNG. Exercises: magic-byte check, chunk parsing via `BinaryFormat.Choice` + length-prefixed payloads, CRC32 verification per chunk, `Binary.Decompress` host call.
2. **All five filter types (None / Sub / Up / Average / Paeth).** Each filter is a separate numeric path applied per scanline. Paeth predictor in particular stresses integer arithmetic, absolute difference, and conditional selection.
3. **RGB and RGBA, 8-bit.** Adds per-channel iteration and confirms scanline filter operates on bytes, not samples.
4. **Palette (PLTE + optional tRNS).** Tests cross-chunk state — palette read from one chunk used during pixel emission from another.
5. **Packed bit depths (1/2/4 bpp).** Byte-aligned per scanline but sub-byte within. Tests bit-level unpacking after decompression.
6. **16-bit depths.** Big-endian sample reads through `BinaryFormat.UnsignedInteger16` with `ByteOrder.BigEndian`.
7. **Adam7 interlacing.** Seven sub-images reassembled into the final raster. Tests iteration patterns and list reassembly logic.
8. **Negative cases** (corrupt CRC, bad filter byte, truncated IDAT). Must surface as M errors at the right place; tests `try`/`otherwise` and error record metadata.

### Test corpus

[PngSuite](http://www.schaik.com/pngsuite/) by Willem van Schaik. Few hundred KB total. Exercises every combination of bit depth, colour type, interlacing, and ancillary chunks, plus deliberately corrupted files. ~25 years of de facto status as the PNG conformance set.

For each file: decode → produce row-major RGBA8 byte buffer → SHA-256. Compare against a frozen oracle generated by a reference decoder (suggest `image` crate in Rust or `libpng` directly). Store oracle hashes in a sidecar JSON or TOML, version-controlled.

### Diagnostic granularity

Per-chunk CRC verification is the key win — when a file fails to decode, the first failing CRC localises the bug to one chunk boundary. The runner should report per-chunk pass/fail, not just final-image pass/fail.

## Part 2: PQTI (Power Query Test Image) Track

### Format design goals

1. Trivial to parse with `BinaryFormat`. Parsing bugs must not mask evaluation bugs.
2. Each section targets a specific M evaluator path.
3. Self-describing: file contains its own expected pixel hash and metadata.
4. Output is an image, so failures are also eyeball-able.
5. The pixel buffer is *computed* from prior section results, not stored. Any upstream evaluation bug → wrong pixels → wrong hash. One uniform pass/fail criterion across the whole suite.

### Wire format

All multi-byte integers big-endian (matches `BinaryFormat` defaults for image work).

```
PQTI File Layout:
+----------------------------------+
| Magic         : "PQTI" (4 bytes) |
| Version       : u16              |
| Flags         : u16              |
| Section count : u16              |
| Reserved      : u16 (zero)       |
+----------------------------------+
| Section 0                        |
| Section 1                        |
| ...                              |
| Section N-1  (pixel program)     |
+----------------------------------+
| Trailer:                         |
|   Expected pixel hash : 32 bytes |
|   (SHA-256 of row-major RGBA8)   |
+----------------------------------+

Each section (TLV):
  tag    : u16
  length : u32
  payload: bytes[length]
```

TLV with length prefix means a decoder can skip unknown section tags. This is the forward-compatibility lever: add new test sections without breaking old decoders.

### Section tags (initial set)

| Tag    | Name                | Purpose                                                                                                                |
|--------|---------------------|------------------------------------------------------------------------------------------------------------------------|
| 0x0001 | `HEADER`            | Width, height, image colour mode (RGB8/RGBA8), human-readable test name, feature-flag bitmap.                          |
| 0x0010 | `NUMERIC`           | List of `(opcode, a, b, expected)` tuples. Covers overflow at 2³¹/2⁵³, div-by-zero, neg modulo, int/float coercion.    |
| 0x0011 | `BITWISE`           | Bitwise op test vectors. Separate from `NUMERIC` to isolate `Number.BitwiseAnd/Or/Xor/Shift` from arithmetic.          |
| 0x0020 | `LIST_SHAPES`       | Instructions to build lists of specified shapes (deep nest, long flat, lazy-then-take). Each carries an expected hash. |
| 0x0021 | `RECORD_TABLE`      | Embedded records/tables across every M primitive type. Round-tripped through sort/filter/group with expected output.   |
| 0x0030 | `ERROR_PATHS`       | Triggers errors deliberately. Each entry: `(operation, must_error: bool, expected_error_reason)`. Tests `try`.         |
| 0x0040 | `TEXT_ENCODING`     | UTF-8 byte sequences (incl. surrogate-pair-equivalent paths, BOM, invalid). Tests `Text.*` correctness.                |
| 0x00F0 | `PIXEL_PROGRAM`     | The drawing program. Always the last data section before the trailer.                                                  |

Tags 0x0000–0x00FF reserved for core. 0x0100+ available for project-specific extensions.

### The pixel program (tag 0x00F0)

A short bytecode that the decoder executes to produce the final pixel buffer. Instructions deliberately reference results computed by earlier sections, so any upstream miscalculation propagates to wrong pixels.

Instruction set (initial):

| Opcode | Mnemonic           | Operands                                | Effect                                                                  |
|--------|--------------------|-----------------------------------------|-------------------------------------------------------------------------|
| 0x01   | `FILL`             | r, g, b, a                              | Fill entire buffer.                                                     |
| 0x02   | `RECT`             | x, y, w, h, r, g, b, a                  | Fill rectangle.                                                         |
| 0x03   | `HGRAD`            | x, y, w, h, r0,g0,b0,a0, r1,g1,b1,a1    | Horizontal linear gradient.                                             |
| 0x04   | `VGRAD`            | (as above)                              | Vertical linear gradient.                                               |
| 0x10   | `FROM_NUMERIC`     | section_index, x, y                     | Write a colour computed from a numeric-section result at `(x, y)`.      |
| 0x11   | `FROM_LIST_HASH`   | section_index, list_index, x, y, w, h   | Use a hash of a list-section result as RGBA seed; fill rect.            |
| 0x20   | `XOR_REGION`       | x, y, w, h, r, g, b, a                  | XOR a constant colour over a region (exercises bitwise + iteration).    |
| 0xFF   | `END`              | —                                       | Program terminates.                                                     |

The point is not graphical interest — it's that the pixel buffer is a deterministic function of every prior section's correct evaluation.

### Suite organisation

Many small files (~1 KB each) rather than one large file. Each file isolates one or two evaluator concerns:

```
suite/
├── numeric_overflow_i32.pqti
├── numeric_overflow_i53.pqti
├── numeric_div_by_zero.pqti
├── numeric_neg_modulo.pqti
├── list_deep_recursion.pqti
├── list_lazy_take.pqti
├── list_long_flat.pqti
├── record_all_primitives.pqti
├── error_try_otherwise.pqti
├── error_uncaught_propagation.pqti
├── bitwise_shift_boundaries.pqti
├── text_utf8_edge.pqti
└── ...
```

Test runner iterates the directory, decodes each, compares the computed pixel hash to the trailer hash, reports pass/fail. Detailed per-section diagnostics emitted on failure.

### Adversarial design notes

The format should be easy to *parse* but designed so the *test content* probes known fragile areas. Specifically include:

- A list-shape entry that materialises right at the recursion ceiling you've set.
- A numeric entry that computes `2^53 + 1` and verifies it does *not* equal `2^53` (catches silent f64 demotion).
- An error-paths entry that wraps a guaranteed-error operation in `try` and asserts the error record's `Reason` field. Tests both that errors fire and that error records carry the right metadata.
- A bitwise entry shifting by 32 and by 64 on a 32-bit value. Defined behaviour in M but a common bug site in evaluators.
- A text entry containing valid UTF-8 that decodes to characters outside the BMP. Surrogate handling is a classic bug surface.

When the evaluator gains a new feature or fixes a bug, add a PQTI file that would have caught it. The suite grows with the project.

## Part 3: Rust Generator

A standalone Rust binary, `pqti-gen`, that emits PQTI files from high-level descriptions.

### Why Rust

Matches the evaluator's implementation language. Two independent implementations of the same spec (Rust generator + M decoder) cross-check each other — if both agree on the expected hash and the actual decoded hash matches, three implementations agree. If the M decoder disagrees with the Rust generator's claimed hash, *one of them is wrong* and you need to disambiguate with a reference Power Query evaluator.

### Structure

```
pqti-gen/
├── src/
│   ├── main.rs          # CLI: pqti-gen <test-descriptions-dir> <output-dir>
│   ├── format.rs        # PQTI wire format encoding
│   ├── sections/        # One module per section tag
│   │   ├── header.rs
│   │   ├── numeric.rs
│   │   ├── list_shapes.rs
│   │   ├── ...
│   │   └── pixel_program.rs
│   ├── interpreter.rs   # Executes the pixel program to compute expected hash
│   └── hash.rs          # SHA-256 wrapper
└── tests/
    └── roundtrip.rs     # Generate → parse-back → assert equality
```

Test descriptions in TOML, one per output file:

```toml
# numeric_overflow_i32.pqti.toml
name = "Numeric: 32-bit integer overflow boundaries"
width = 32
height = 32
mode = "RGBA8"

[[numeric]]
op = "add"
a = 2147483647     # i32::MAX
b = 1
expected = 2147483648   # M numbers are 64-bit; no overflow expected

[[numeric]]
op = "mul"
a = 65536
b = 65536
expected = 4294967296

[[pixel_program]]
op = "FILL"
args = [128, 128, 128, 255]
```

The generator parses TOML, builds the binary PQTI file, computes the expected pixel hash by running its own pixel-program interpreter, and writes the hash into the trailer.

### Cross-check workflow

Per project decision: expected results are produced by both the Rust generator (first principles) and a reference Power Query evaluator (Excel desktop or Power BI). The flow:

1. Rust generator emits PQTI files with its own computed expected hashes.
2. A separate harness runs the M decoder under a reference Power Query evaluator and records its actual hashes.
3. CI asserts: reference-evaluator-hash == rust-generator-hash. Disagreement means either the generator or our understanding of M semantics is wrong; investigate before declaring the file ready for evaluator testing.
4. Once Rust and reference agree, the file is canonical. The Rust evaluator under test must match.

This makes the reference evaluator a one-time oracle per test file rather than a continuous dependency.

## Part 4: M Decoder Structure

### File layout

Both decoders live in the same M module structure to share common helpers.

```
m-decoders/
├── Common/
│   ├── BitReader.pq        # Byte-level + sub-byte bit reader (PNG packed depths, etc.)
│   ├── Hash.pq             # SHA-256 in M (or host-bound if available)
│   ├── CRC32.pq            # Table-driven CRC32
│   └── PixelBuffer.pq      # Row-major RGBA8 buffer helpers
├── PNG/
│   ├── Decode.pq           # Entry point
│   ├── Chunks.pq           # BinaryFormat-based chunk parser
│   ├── Filters.pq          # Five scanline filters
│   ├── Unpack.pq           # Bit-depth unpacking
│   ├── Adam7.pq            # Interlace reassembly
│   └── Palette.pq
└── PQTI/
    ├── Decode.pq           # Entry point — returns a record per the test harness contract
    ├── Sections.pq         # BinaryFormat for TLV sections
    ├── Numeric.pq
    ├── Lists.pq
    ├── ErrorPaths.pq
    └── PixelProgram.pq     # Bytecode interpreter
```

### M decoder contract

Both decoders expose a single entry function:

```
Decode = (input as binary) as record =>
    [
        Success = ...,           // logical
        PixelHash = ...,         // text (hex SHA-256)
        ExpectedHash = ...,      // text (from file trailer for PQTI, from oracle for PNG)
        Sections = ...,          // table: section name, expected, actual, status
        Errors = ...             // list of error records, empty if Success
    ]
```

The harness:

1. Calls `Decode` on each file.
2. Asserts `Success = true` and `PixelHash = ExpectedHash`.
3. On failure, dumps the `Sections` table and `Errors` list.

### Implementation order

Mirror the PNG coverage progression. PQTI sections in tag order, simplest first (`HEADER` → `NUMERIC` → `BITWISE` → `PIXEL_PROGRAM` with `FILL`/`RECT` only → harder list/error/text sections → richer pixel-program opcodes).

The M decoders should be *boring* — straightforward use of `BinaryFormat` and standard M. They are the system under test indirectly: bugs in the evaluator will manifest as decoder failures, but the decoder code itself should not be clever.

## Open questions for the implementing instance

1. **Hash function in M.** Is SHA-256 callable from the evaluator (host binding) or do we implement it in M? If implementing in M, that's itself a major test — but it makes the pixel-hash check circular if the SHA-256 implementation has bugs. Suggest: host-bind it, and have a separate small M-native SHA-256 implementation used only for cross-checking the host binding.
2. **CI integration.** Where does the reference-evaluator oracle run? Manual one-time generation, or scheduled CI job against an Excel/Power BI instance?
3. **Performance budget.** What's acceptable per-file decode time on the evaluator? Sets a ceiling on PQTI file sizes and on PNG image dimensions in the suite.
4. **Error record shape.** PQTI's `ERROR_PATHS` section asserts on error record fields. Confirm the exact field names and types the evaluator produces (`Reason`, `Message`, `Detail`?) so the spec freezes them.
5. **Endianness of PQTI.** Spec'd as big-endian above for symmetry with PNG/JPEG. Confirm this is fine; little-endian would match Rust's native and shave a few `BinaryFormat` calls.

## Suggested first milestone

A single PQTI file, `smoke.pqti`, containing only:

- `HEADER` (16×16 image, mode RGBA8, name "smoke")
- `PIXEL_PROGRAM` with one `FILL 0xFF 0x00 0x00 0xFF` instruction
- Trailer with the SHA-256 of a 16×16 solid-red RGBA8 buffer

Plus the M decoder for those two sections and the harness. This proves the wire format works end-to-end before any test content is added. From there, each subsequent PR adds one section type and a handful of test files exercising it.

---

## Non-goals

- This is not an image-processing project. The image is a hash oracle, nothing more.
- The PQTI format is not intended for any use outside this test suite. Do not over-engineer it.
- We are not writing a JPEG decoder. JPEG was considered and rejected: too much complexity (three entropy coders, four SOF modes, bit-packed Huffman streams) for the evaluator surface it would exercise.

## Deferred future work: visual output in Excel

**Status: deferred until the core decoder is debugged and passing.** Noting here so the idea isn't lost. Do not implement until the hash-based test suite is green.

Once the decoders work, the M output can additionally render as a table of Unicode quadrant block glyphs (U+2580–U+259F) such that the table displayed in an Excel sheet *is* the image. A H×W pixel image becomes a (H/2)×(W/2) table where each cell holds one of 16 quadrant glyphs encoding which sub-pixels of its 2×2 block are "on".

Optional colour extension: parallel tables of foreground/background RGB applied to cells via conditional formatting give two-colours-per-2×2-block — recognisable colour images at modest sizes.

When this lands, PQTI test files should be designed so their pixel programs produce recognisable images (e.g. `numeric_overflow_i32.pqti` decodes to pixel-rendered "i32"). The pass criterion remains the hash, but a passing suite produces a visually correct workbook and a failing test produces a visibly wrong picture — diagnostic at a glance.

This change adds optional fields to the decoder contract:

```
GlyphTable    = ...,   // table of text, H/2 × W/2
ForegroundRGB = ...,   // optional, parallel to GlyphTable
BackgroundRGB = ...,   // optional, parallel to GlyphTable
```

Existing fields (`PixelHash`, `ExpectedHash`, `Sections`, `Errors`) remain unchanged.
